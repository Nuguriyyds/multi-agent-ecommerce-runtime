# 智能电商推荐系统

## 产品目标

**解决了什么问题：**传统电商三大痛点
- 痛点：推荐结果和库存脱节，推荐了缺货商品；本项目做法：库存实时校验，缺货自动剔除
- 痛点：营销文案千篇一律，所有人看同一段广告语；本项目做法：根据用户画像生成个性化文案
- 痛点：各系统各自为战，推荐、文案、库存三套系统互不感知；本项目做法：统一编排，结果实时互相影响

## 用户角色

**终端消费者：**浏览商品时，系统自动返回个性化推荐+文案；失败情况：推荐商品不准、推荐了缺货商品、用户不感兴趣、看到报错页面……
**运营人员：**通过 API 或日志查看推荐效果、配置 A/B 实验参数；失败情况：无法正常看到日志……
**开发者：**通过 API 接入推荐能力，集成到现有电商系统；失败情况：接口不稳定、响应超时……

## 子问题

一个请求进来（给我推荐商品），系统需要做四件事：

1. **了解用户（画像 Agent）**：拿到用户 ID → 从行为数据库查历史（浏览/点击/购买）→ **调 LLM 分析行为数据** → 输出结构化用户画像（分群标签、品类偏好、价格区间）
2. **召回+排序商品（推荐 Agent）**：分两步——**粗召回**从商品库捞候选（不用 LLM，快），**精排**让 LLM 根据用户画像对候选重排（要 LLM，慢）
3. **校验库存（库存 Agent）**：检查候选商品库存状态，剔除缺货商品，标记低库存预警。**不调 LLM**，纯数据校验
4. **生成文案（文案 Agent）**：根据用户分群选 prompt 模板 → **调 LLM 生成** → 广告法合规过滤

**为什么用 Agent 而不是普通函数？** 画像、推荐、文案三个步骤都调 LLM 做 AI 推理（分析用户行为、判断商品匹配度、创作个性化文案），不是写死的 if-else 规则。LLM 的推理能力是"智能"的来源。Agent 架构给每个 LLM 调用加了统一的超时、重试、降级保护。

**执行顺序（3 阶段编排）**：

```
Phase 1（并行）:
  ├─ 画像 Agent (LLM): 分析用户行为 → UserProfile
  └─ 推荐 Agent 粗召回 (无LLM): 从商品库捞 num_items×3 候选

Phase 2（并行）:
  ├─ 推荐 Agent 精排 (LLM): 拿到画像后，LLM 对候选重排 → Top-N
  └─ 库存 Agent (无LLM): 校验候选库存 → 可用商品列表

Supervisor 中间处理: 用库存过滤精排结果 → 截断到 num_items

Phase 3（串行）:
  └─ 文案 Agent (LLM): 根据画像+最终商品 → 个性化文案
```

**为什么是 3 阶段而不是"全部并行"？**
- 精排需要画像（必须等 Phase 1 的画像 Agent 完成）
- 文案需要最终商品列表（必须等库存过滤后才知道推荐哪些商品）
- 粗召回不需要画像，所以能和画像并行（Phase 1）
- 精排和库存互不依赖，所以能并行（Phase 2）
- 这样总耗时 ≈ 3 次 LLM 调用（但其中两次并行了），比全串行快约 40%

**失败处理**：不同步骤的严重程度不一样
- 了解用户失败 → 画像为空，推荐跳过精排，走冷启动路径（热门商品）
- 库存校验失败 → **严重**，推荐缺货商品是业务事故，Agent 内部走缓存兜底
- 文案生成失败 → 不严重，Agent 内部返回商品默认描述
- 粗召回失败 → 核心功能，无法兜底，返回错误

## 技术选型

| 选什么 | 选了什么 | 为什么选它，不选另一个 |
|--------|---------|---------------------|
| 并行方案 | asyncio（`asyncio.gather()`） | Agent 全是 IO 密集（等 LLM、等库存查询），不做 CPU 计算。asyncio 单线程无并发安全问题，比多线程轻量；多进程开销太大，不适合 IO 场景 |
| Web 框架 | FastAPI | 原生支持 async（Agent 是 async 的，框架必须匹配）；内置 Pydantic 校验和自动 API 文档。Flask 默认同步，Django 太重 |
| 数据传递 | Pydantic 模型 | Agent 之间传 dict 的话，字段缺失或类型错误要到运行时才发现；Pydantic 在数据创建时就校验，错误不会悄悄传播到下游 |
| LLM 调用 | openai SDK（AsyncOpenAI） | 3 个 Agent 调 LLM，但都只用 chat completion。langchain-openai 引入三个重依赖（langgraph + langchain + langchain-openai），只用了最基础的"发消息-收回复"功能，不值得。openai SDK 一个包就够了 |
| Agent 编排 | 自己写 Supervisor（不用 LangGraph） | 只有 4 个 Agent，3 阶段 DAG；自己写能完全控制超时、重试、降级策略，面试时能讲清楚每一行 |
| 重试机制 | BaseAgent 自己实现（指数退避） | F02 已经做好了，不额外引入 tenacity |
| 日志 | 标准库 logging | 不引入 structlog，logging + 自定义 Formatter 也能输出 JSON |
| A/B 实验 | random.betavariate (Thompson Sampling) | 不引入 numpy，标准库够用 |
| 外部服务 | 全部 mock（内存实现） | 首期不依赖 Redis/Milvus/MySQL，mock 分布在 `app/v1/services/` 与 `app/shared/data/`，接口相同，后续可替换 |

## 架构图

```
用户请求 (POST /api/v1/recommend)
   │
   ▼
API 接口层（FastAPI）
   │  收请求 + 生成 trace_id + 返结果
   ▼
A/B 测试引擎 → 给用户分实验组
   │
   ▼
Supervisor（编排器）
   │
   ├── Phase 1: asyncio.gather() ─┬── 画像 Agent (LLM)
   │                               │     行为数据 → LLM 分析 → UserProfile
   │                               │     失败 → profile=None，走冷启动
   │                               │
   │                               └── 推荐 Agent 粗召回 (无LLM)
   │                                     商品库 → 基础排序 → 3x 候选
   │
   ├── Phase 2: asyncio.gather() ─┬── 推荐 Agent 精排 (LLM)
   │                               │     画像 + 候选 → LLM 重排 → Top-N
   │                               │     无画像 → 跳过精排，用粗召回结果
   │                               │
   │                               └── 库存 Agent (无LLM)
   │                                     候选 → 库存校验 → 可用列表
   │                                     失败 → 走缓存兜底
   │
   ├── 中间处理: 库存过滤 + 截断到 num_items
   │
   └── Phase 3: ─────────────────── 文案 Agent (LLM)
                                      画像 + 商品 → 选模板 → LLM 生文案
                                      失败 → 商品默认描述
   │
   ▼
  聚合 RecommendationResponse → 返回 JSON
```

## 数据模型

### 领域模型（`app/shared/models/domain.py`）

```
UserSegment (枚举): NEW_USER / ACTIVE / HIGH_VALUE / PRICE_SENSITIVE / CHURN_RISK

UserProfile: user_id, segments, preferred_categories, price_range, rfm_score, tags, cold_start
Product: product_id, name, category, price, description, brand, stock, tags, score, image_url
InventoryStatus: product_id, available, stock, low_stock, purchase_limit
MarketingCopy: product_id, copy_text
RecommendationRequest: user_id, scene, num_items
RecommendationResponse: request_id, user_id, products, marketing_copies, experiment_group, agent_details, latency_ms
```

### Agent IO 类型（`app/v1/models/agent_io.py`）

每个 Agent 的 execute() 内部用 Pydantic 做入口校验 + 出口序列化，BaseAgent 的 dict 契约不变：
```python
async def execute(self, input_data: dict) -> dict:
    inp = UserProfileInput(**input_data)           # 入口校验
    # ... 业务逻辑 ...
    return UserProfileOutput(profile=...).model_dump()  # 出口序列化
```

## 服务层

| 服务 | 文件 | 职责 |
|------|------|------|
| LLMService | `app/v1/services/llm_service.py` | openai AsyncOpenAI 封装；API key 为空走 mock（返回确定性模板数据） |
| FeatureStore | `app/v1/services/feature_store.py` | 用户行为数据（内存 dict，预置 5 个样本用户，每个对应一个分群） |
| ProductCatalog | `app/shared/data/product_catalog.py` | 商品库（内存 list，15-20 个商品，涵盖 6-8 个品类） |
| InventoryStore | `app/shared/data/inventory_store.py` | 库存数据 + 缓存兜底（成功时更新缓存，失败时返回缓存） |
| ABTestEngine | `app/v1/services/ab_test.py` | 一致性哈希分桶 + Thompson Sampling 动态流量分配 |
| MetricsCollector | `app/v1/services/metrics.py` | Agent 调用次数/成功率/平均延迟统计 |

全部直接实现 mock 版本，不提前建 ABC 抽象层（YAGNI）。等需要接真实服务时再抽接口。

**F03 用户画像 Agent**

- 做什么：FeatureStore 取行为数据 → 构建 prompt → LLM 分析 → 解析 JSON → UserProfile。冷启动用户不调 LLM，直接返回 cold_start=true
- 验收：有历史→结构化画像；无历史→冷启动；pytest 全过

**F04 商品推荐 Agent**
- 创建：`app/shared/data/product_catalog.py`, `app/v1/agents/product_rec.py`, `tests/v1/test_product_rec_agent.py`
- 做什么：两种模式——粗召回（无 profile → 基础排序，不调 LLM）和精排（有 profile → LLM 根据用户偏好重排候选）
- 验收：粗召回返回候选；精排返回个性化排序；冷启动走粗召回

**F05 库存校验 Agent**
- 创建：`app/shared/data/inventory_store.py`, `app/v1/agents/inventory.py`, `tests/v1/test_inventory_agent.py`
- 做什么：不调 LLM。查库存 → stock=0 剔除 → 低库存预警 → 限购策略。失败走缓存兜底
- 验收：零库存过滤；低库存预警；故障缓存兜底

**F06 营销文案 Agent**
- 创建：`app/v1/agents/marketing_copy.py`, `tests/v1/test_marketing_copy_agent.py`
- 做什么：按分群选 prompt 模板（5 种风格）→ LLM 生成 → 广告法合规过滤。失败返回默认描述
- 验收：不同画像→不同风格；敏感词被过滤；超时→默认描述

**F07 Supervisor 编排器**
- 创建：`app/v1/orchestrator/supervisor.py`, `tests/v1/test_supervisor.py`
- 做什么：3 阶段管线，调 4 个 Agent（推荐 Agent 调两次），聚合为 RecommendationResponse
- 验收：端到端完整结果；模拟超时不影响其他 Agent；并行耗时≈max(并行 Agent)

**F08 A/B 测试引擎**
- 创建：`app/v1/services/ab_test.py`, `tests/v1/test_ab_test.py`
- 做什么：一致性分桶 (MD5 hash) + Thompson Sampling。默认实验：rec_strategy (control vs treatment)
- 验收：同用户→同桶；分布均匀；Thompson Sampling 收敛

**F09 API 层**
- 修改：`app/main.py`, `app/shared/config/settings.py`；创建：`tests/v1/test_api.py`

- 验收：curl 返回完整 JSON；冷启动用户不报 500；trace_id 在响应头

**F10 LLM 重排增强**

**F11 演示数据 + 冒烟测试**

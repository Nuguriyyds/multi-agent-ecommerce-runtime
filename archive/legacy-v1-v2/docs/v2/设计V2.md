## 1. 产品目标

### V1 回顾

V1 是一个推荐 API：客户端发 `POST /recommend`，后端按固定 3 阶段管线（画像 → 选品+排序+库存 → 文案）返回推荐结果。无对话能力，无偏好持久化，无推荐反馈闭环。

### V2 目标

V2 不再只是推荐 API，而是一个**面向大型电商系统的多入口 Agent 平台**。一期目标不是做“单个聊天机器人”，而是打通：

`conversation(advisory-first) -> session memory -> profile projection -> homepage recommendation projection -> snapshot read`

### V2.1 Delta（中心化 Agent Runtime 补完整度）

- 以下条目描述的是 V2.1 已落地内容；若与下文 V2.2 delta 冲突，以 V2.2 为准。
- V2.1 继续保持“用户只与主 agent 交互、固定 worker 池互不通信、worker 只返回结构化结果”的中心化模型。
- planner 是规则驱动 planning，不是真实自由 LLM planner；可用 step 固定为 `clarify / preference_worker / catalog_worker / inventory_worker / comparison_worker / copy_worker / recommendation.request_refresh`。
- 每轮 `/messages` 会先生成 `TurnPlan`，再执行 worker 与 tool，并把父任务、worker 任务、tool 任务统一写入 `task_records`。
- 新增 debug trace API：`GET /api/v2/sessions/{session_id}/turns/{user_turn_number}/trace`，V2.1 查看 `plan / tasks / refresh`，V2.2 已升级为 `plan / tasks / projection`。
- feedback 已从“只入库”升级为最小闭环：写入 `click / skip / purchase` 后会生成 `feedback summary`，并驱动对话选品、后台 refresh、按需 recommendation read 的排序。

### V2.2 Delta（导购优先聊天 + 后台画像投影 + 纯快照推荐）

- V2.2 继续保持中心化模型：用户只与 `ShoppingManager` 交互，固定 worker 池互不通信，worker 只返回结构化结果。
- planner 仍是规则驱动，不是真实自由 LLM planner；聊天意图固定分流为 `clarify / fallback / advisory / recommendation / comparison`。
- `/messages` 默认只做导购、澄清和偏好收集；只有明确推荐 / 比较意图时才返回 `products / comparisons / copies`。
- `/messages` 同步只写 `session memory`；长期 `UserProfile` 改为后台 `profile_projection` 事件生成。
- 后台继续复用 `events + task_records`，以 `profile_projection -> recommendation_refresh(homepage)` 的链路生成快照。
- `GET /api/v2/users/{user_id}/recommendations` 改为纯快照读取，不再同步现场生成 snapshot；miss / expired 只返回空值或旧值，并异步入队 `homepage` refresh。
- trace 语义从 `refresh` 改为 `projection`，用于说明本轮是否发起了后台投影请求。

### 系统形态

#### C 端：对话购物助手

- 用户与 AI 助手对话，表达需求、预算、品牌偏好、用途场景、排除项。
- `ShoppingManager` 在对话中逐步收集偏好，实时调度选品、比较、库存、文案等 worker 与 tool。
- 用户只与主 manager 交互，不直接接触底层 worker。

#### 推荐侧：后台个性化服务

- 对话过程中产生的结构化信号先同步回写 `session memory`。
- 当偏好稳定或被修正时，系统只写入 `profile_projection` 事件；后台再生成 `UserProfile`，并继续触发 `homepage` 的 `recommendation_refresh`。
- 首页、商品页、购物车页优先消费已有推荐快照；`/recommendations` 不再依赖同步现场生成。

#### B 端：运营 / 商家助手

- B 端与 C 端**不冲突**，但不与 C 端共用同一个人格化 agent。
- 上层入口 agent 分开，底层共享 `Tool / TaskBoard / Memory / Event / State` 基础设施。
- 一期只预留 `merchant_ops` manager 注册位，不实现 B 端业务入口。

### 目标链路

```text
用户与购物助手对话
  → 主 manager 抽取预算、品类、品牌偏好、场景、拒绝项
  → 信号同步写入 session memory
  → 偏好稳定或修正时写入 profile_projection 事件
  → 后台生成 user profile，并继续触发 homepage recommendation_refresh
  → recommendation snapshot 进入首页 / 商品页 / 购物车读取链路
  → 用户点击、跳过、购买再反哺后续 snapshot 投影
```

## 2. 系统边界

### 一期 scope

| 做 | 不做 |
|----|------|
| C 端对话购物助手 | 匿名用户画像合并 |
| 推荐闭环：画像更新 + 异步刷新 + 快照消费 | 独立消息队列 / 独立 worker 进程 |
| 多 Manager 的共享 runtime 底座 | B 端运营/商家业务入口 |
| `shopping` manager 落地，`merchant_ops` 预留 | 真实 MCP server |
| SQLite 持久化（WAL） | 正式 Scheduler 子系统 |
| 内部 MCP-compatible ToolRegistry | 语音、图片、视频等多模态输入 |
| 进程内后台任务 | 完整反馈学习闭环 |

### 用户假设

- 用户已登录，`user_id` 由客户端传入。
- 一期只处理登录用户。
- 一期内 `scene` 采用显式上下文策略：
  - `default`、`homepage` 可仅依赖用户画像
  - `product_page` 必须带 `product_id`
  - `cart` 必须带 `product_ids`

## 3. 平台架构

### 3.1 总体结构

系统采用**多 Manager、共享 Runtime、薄领域 Worker** 的分层结构：

1. **Manager Layer**
   - 面向用户或业务入口。
   - 当前只实现 `ShoppingManager`。
   - 后续预留 `merchant_ops` manager。

2. **Runtime Layer**
   - `Policy Gate`
   - `Hook Bus`
   - `PromptRegistry`
   - `TaskBoard`
   - `Memory`
   - `EventProcessor`
   - `Snapshot Store`

3. **Worker Layer**
   - `preference_worker`
   - `catalog_worker`
   - `comparison_worker`
   - `inventory_worker`
   - `copy_worker`

4. **Tool / Data Layer**
   - `session.*`
   - `profile.*`
   - `catalog.*`
   - `inventory.*`
   - `copy.*`
   - `recommendation.*`
   - `feedback.*`

### 3.2 关键原则

- 只有主 manager 与用户交互。
- worker 之间**禁止直接通信**。
- worker 只能被主 manager 调度，并向主 manager 返回结构化结果。
- 共享 runtime 是平台中心；legacy V1 代码只能作为实现来源，不再是顶层编排中心。

### 3.3 架构图

```text
                         ┌────────────────────────┐
                         │         用户            │
                         └──────────┬─────────────┘
                                    │ HTTP
                         ┌──────────▼─────────────┐
                         │        FastAPI          │
                         │   /api/v2/sessions/*    │
                         └──────────┬─────────────┘
                                    │
                      ┌─────────────▼──────────────┐
                      │      ManagerRegistry        │
                      │  shopping / merchant_ops    │
                      └─────────────┬──────────────┘
                                    │
                         ┌──────────▼──────────┐
                         │   ShoppingManager    │
                         │ - 理解意图           │
                         │ - 规划 worker tasks  │
                         │ - 组织澄清与回复     │
                         └─────┬────────┬──────┘
                               │        │
                   ┌───────────▼───┐ ┌──▼─────────────────┐
                   │  Policy Gate   │ │     Hook Bus       │
                   └───────────┬────┘ └──┬─────────────────┘
                               │          │
                  ┌────────────▼──────────▼────────────┐

                ┌──────────────▼──┐   ┌──▼──────────────┐

                      │    │               │    │
         preference_worker  ...       session.*  profile.* ...
```

### 3.4 主 manager 与 worker 的通信边界

#### 主 manager 负责

- 接收用户输入与会话上下文。
- 进行 policy 判断：`allow / reject / clarify`。
- 决定是否需要追问、选品、比较、库存查询、文案生成。
- 生成 worker task，并收集 worker 结果。
- 组织最终用户回复。
- 在偏好稳定后触发后台推荐刷新。

#### worker 负责

- 处理单一领域任务。
- 调用本 worker 允许使用的工具。
- 返回结构化结果，不返回人格化对话内容。
- 不直接依赖其他 worker。

### 3.5 单轮执行循环

```text
收到用户消息
  → 读取 SessionState + UserProfile + 最近场景上下文
  → Policy Gate 判断 allow / reject / clarify
  → ShoppingManager 生成 TurnPlan（clarify / fallback / advisory / recommendation / comparison）
  → 顺序执行 worker（必要时调用 tool）
  → 收集 WorkerResult
  → 更新 session memory / runtime scratchpad / task records
  → 判断 terminal_state:
      - reply_ready
      - needs_clarification
      - fallback_used
  → 如偏好稳定或被修正，写 profile_projection 事件
  → recommendation / comparison 意图按需返回商品结果；默认 advisory 仅返回导购回复
  → 返回用户回复
```

## 4. Runtime 内核

### 4.1 Policy Gate

首版采用轻量三态：

- `allow`：允许继续执行。
- `reject`：非法请求、未支持能力、越权动作、非法工具、明显错误参数。
- `clarify`：信息不足但可以继续，例如：
  - 比较商品数不足
  - `product_page` 缺 `product_id`
  - `cart` 缺 `product_ids`
  - 预算、品类等关键信息缺失

首版不实现用户审批流，不引入 `ask_user_confirmation`。

### 4.2 Hook Bus

首版只实现核心 hook 集，不做全量事件框架：

- `turn.started`
- `turn.finished`
- `worker.started`
- `worker.finished`
- `worker.failed`
- `tool.before`
- `tool.after`
- `tool.error`
- `profile.updated`
- `snapshot.refreshed`
- `background_task.started`
- `background_task.finished`
- `background_task.failed`

Hook 约束：

- 输入为当前上下文快照。
- 输出为结构化结果。
- Hook 不能直接篡改 runtime 对象，只能通过主系统允许的返回字段影响流程。

### 4.3 Prompt System

首版采用集中注册且版本化的 `PromptRegistry`：

- 每个 prompt 具备 `name / version / template / variables_schema`。
- manager 与 worker 只能从 registry 读取 prompt，不在业务代码中散落硬编码模板。

首版至少包含以下 prompt：

- `shopping.manager.plan`
- `shopping.manager.clarify`
- `shopping.manager.respond`
- `preference.extract`
- `comparison.summarize`
- `copy.generate`

### 4.4 Task System

TaskBoard 首版统一记录两类任务：

1. **会话任务**
   - 主 manager 在单轮内调度的 worker task
2. **后台任务**
   - 推荐刷新等异步任务

每条任务都需要记录：

- `task_id`
- `task_scope`
- `session_id`
- `turn_id`
- `event_id`
- `manager_name`
- `worker_name`
- `tool_name`
- `step`
- `status`
- `input`
- `output`
- `error`
- `latency_ms`

### 4.5 后台任务系统

V2.2 中后台继续复用 `events + task_records`，但核心任务已扩展为两类：

```text
对话中偏好稳定 / 偏好修正
  → 写 profile_projection event
  → EventProcessor 轮询事件
  → 生成或更新 user profile
  → 写 homepage recommendation_refresh event
  → 读取最新 user profile + feedback summary
  → 生成或更新 homepage snapshot
  → 写 task record
  → 标记事件完成/失败
```

### 4.6 Scheduler 预留

- 首版不实现正式 Scheduler 子系统。
- 仅预留接口，后续可接 TTL 清理、离线 refresh、B 端批量任务。
- 首版任何核心链路都不能依赖定时调度才能成立。

### 4.7 硬护栏

| 护栏 | 规则 | 触发后果 |
|------|------|---------|
| Step 上限 | 每轮最多 8 个 task step | 超限进入 `fallback_used` |
| 工具白名单 | 只能调用 ToolRegistry 中注册的工具 | 非法工具直接拒绝 |
| Worker 边界 | worker 不得直接调用其他 worker | 违规视为架构错误 |
| Schema 校验 | tool 输入输出都必须通过 Pydantic 校验 | 校验失败按 tool 错误处理 |
| 终止状态 | 每轮必须以 `reply_ready / needs_clarification / fallback_used` 结束 | 否则视为 manager 失败 |
| 总超时 | manager 单轮总超时 30s | 超时进入 fallback |

## 5. Tool 与 Worker 模型

### 5.1 Worker 设计

首版固定 5 类薄领域 worker：

| Worker | 作用 | 可用工具（示意） |
|--------|------|-----------------|
| `preference_worker` | 抽取预算、品类、品牌、用途、排除项 | `session.read_memory`, `profile.extract_preferences` |
| `catalog_worker` | 根据画像、场景与 feedback summary 做候选选品 | `catalog.search_products`, `feedback.read_summary` |
| `comparison_worker` | 对多个商品做结构化比较 | `product.compare` |
| `inventory_worker` | 查询库存与可售状态 | `inventory.check` |
| `copy_worker` | 生成个性化商品文案 | `copy.generate` |

worker 返回统一 `WorkerResult`：

- `worker_name`
- `success`
- `payload`
- `signals`
- `clarification`
- `latency_ms`
- `error`

### 5.2 ToolSpec 定义

```python
class ToolSpec(BaseModel):
    name: str
    description: str
    input_schema: dict
    output_schema: dict
    side_effect_level: Literal["none", "session", "persistent"]
```

`side_effect_level` 语义：

- `none`：纯读
- `session`：只影响当前会话
- `persistent`：影响持久化状态或事件

### 5.3 一期工具清单

#### session domain

- `session.read_memory`
- `session.write_memory`
- `session.append_turn`

#### profile domain

- `profile.extract_preferences`
- `profile.read`
- `profile.upsert`
- `profile.request_projection`

#### catalog / product domain

- `catalog.search_products`
- `product.compare`

#### inventory domain

- `inventory.check`

#### copy domain

- `copy.generate`

#### recommendation domain

- `recommendation.request_refresh`

#### feedback domain

- `feedback.record_event`
- `feedback.read_summary`

### 5.4 MCP-compatible 设计

一期不实现真实 MCP server，但工具定义格式与 MCP Tool 协议对齐：

- `name` 使用 `{domain}.{action}` 命名。
- `input_schema` / `output_schema` 可直接导出 JSON Schema。
- 工具统一通过 `ToolRegistry` 注册。
- 后续若接入真实 MCP，只需在外层增加 `MCPToolAdapter`，不改业务工具定义。

## 6. 状态、记忆与存储

### 6.1 三层记忆模型

```text
┌──────────────────────────────────────────────┐
│              Runtime Scratchpad              │
│ 单轮执行态，不跨轮持久化                     │
│ - 当前 step 结果                             │
│ - worker 聚合结果                            │
│ - 当前回复草稿                               │
└──────────────────────────────────────────────┘
                    │
                    ▼
┌──────────────────────────────────────────────┐
│               Session Memory                 │
│ 会话级短期记忆，持久化                       │
│ - budget                                     │
│ - preferred_brands                           │
│ - use_case                                   │
│ - exclusion                                  │
│ - 最近 scene 上下文                          │
│ - 待澄清字段                                 │
└──────────────────────────────────────────────┘
                    │ 达到稳定条件时回写
                    ▼
┌──────────────────────────────────────────────┐
│                User Profile                  │
│ 跨会话长期画像，持久化                       │
│ - preferred_categories                       │
│ - preferred_brands                           │
│ - use_cases                                  │
│ - excluded_terms                             │
│ - price_range                                │
│ - segments                                   │
│ - tags                                       │
└──────────────────────────────────────────────┘
```

### 6.2 偏好信号（PreferenceSignal）

首版偏好抽取只覆盖五类：

- `budget`
- `product_category`
- `brand`
- `use_case`
- `exclusion`

```python
class PreferenceSignal(BaseModel):
    category: Literal["budget", "product_category", "brand", "use_case", "exclusion"]
    value: str
    confidence: float
    source_turn: int
```

### 6.3 稳定偏好判定

回写长期画像必须同时满足：

1. 至少覆盖 2 类偏好信号
2. 每类信号 `confidence >= 0.7`
3. 最近 3 轮内无同类冲突修正

未达到稳定条件：

- 只保留在 `session memory`
- 不污染长期画像
- 可以继续追问或等待后续对话补足

### 6.4 画像更新策略

- `price_range`：新值覆盖旧值
- `preferred_categories / preferred_brands / use_cases / excluded_terms / tags`：去重合并
- 明确修正已有核心偏好时，记为 `preference_corrected`
- 每次画像更新必须写入事件或任务日志，保证可追踪

### 6.5 存储选型

一期使用 **SQLite + WAL**：

- 零运维
- 支持并发读 + 单写
- 易于本地开发与单进程部署
- 后续可替换为 PostgreSQL / MySQL，仅需更换 repository 层

#### `user_profiles`

```sql
CREATE TABLE user_profiles (
    user_id               TEXT PRIMARY KEY,
    preferred_categories  TEXT NOT NULL DEFAULT '[]',
    preferred_brands      TEXT NOT NULL DEFAULT '[]',
    use_cases             TEXT NOT NULL DEFAULT '[]',
    excluded_terms        TEXT NOT NULL DEFAULT '[]',
    price_range           TEXT,
```

## 7. 推荐闭环与场景快照

### 7.1 触发时机

V2.2 中，聊天链路与推荐投影链路已经拆开。后台事件只在以下时机入队：

| 触发条件 | event_type | trigger 值 | 说明 |
|---------|------------|------------|------|
| 偏好稳定 | `profile_projection` | `preference_stable` | 满足稳定投影条件 |
| 偏好修正 | `profile_projection` | `preference_corrected` | 用户明确修改已有核心偏好 |
| feedback 写入 | `recommendation_refresh` | `feedback_event` | 只刷新 `homepage` |
| 快照 miss | `recommendation_refresh` | `snapshot_miss` | 由纯读接口异步入队 |
| 快照过期 | `recommendation_refresh` | `snapshot_expired` | 由纯读接口异步入队 |

### 7.2 刷新流程

```text
对话中检测到稳定偏好 / 偏好修正
  → 同步更新 session memory
  → 写 profile_projection 事件
  → Background worker 消费 profile_projection
  → 生成并持久化 user profile
  → 写 homepage recommendation_refresh 事件
  → Background worker 消费 recommendation_refresh
  → 读取 user profile + feedback summary
  → 生成 homepage recommendation snapshot
  → 标记事件完成
```

### 7.3 场景规则

#### `default`

- 兼容读取语义
- 优先读 `default` 快照，miss 时 fallback 到 `homepage`
- V2.2 不再把它作为默认投影目标

#### `homepage`

- 唯一默认推荐投影目标
- 仅依赖用户画像即可生成
- 无额外上下文要求

#### `product_page`

- 必须带 `product_id`
- 只读已有 contextual snapshot
- miss 时 fallback 到 `homepage`
- 不再同步现场生成 contextual snapshot

#### `cart`

- 必须带 `product_ids`
- 只读已有 contextual snapshot
- miss 时 fallback 到 `homepage`
- 不再同步现场生成 contextual snapshot

#### 缺少上下文时

- 优先由 `Policy Gate` 返回 `clarify`
- recommendation read 场景下可回退读取 `homepage`

### 7.4 快照消费端

`GET /api/v2/users/{user_id}/recommendations` 的行为：

- 如果命中当前 scene 快照且未过期 → 直接返回，`stale=false`
- 如果命中过期快照 → 返回旧 snapshot，`stale=true`，并异步入队 `homepage` refresh
- 如果当前 scene 缺上下文 → recommendation read 走 fallback 语义，不现场生成
- 如果无快照 → 返回空结果，`generated_at=null`、`stale=true`、`pending_refresh=true`
- `default` miss 时 fallback 到 `homepage`
- `product_page / cart` miss 时只 fallback 到 `homepage`，不现场生成 contextual snapshot

快照默认 TTL：24 小时（可配置）。

### 7.5 Feedback 事件

V2.2 中，feedback 继续保留最小闭环，但语义进一步收口：

- 支持 `click / skip / purchase`
- feedback 先写入 `events` 表
- 系统聚合最近 50 条已完成 feedback，生成 `FeedbackSummary`
- `catalog_worker` 与后台 `recommendation_refresh` 会读取 `feedback summary`
- feedback 写入后只为 `homepage` 触发 `recommendation_refresh`
- feedback 不直接改写 `UserProfile` 主字段，不做在线学习模型

## 10. 兼容与 Legacy 复用

### 10.1 原则

- V2 的顶层架构中心是 `Shared Runtime + ShoppingManager`。
- legacy V1 代码可复用，但只作为实现来源。
- B 端与 C 端入口独立，但底层共享 runtime。

### 10.2 兼容策略

- V2 新接口走 `/api/v2/` 前缀。
- legacy 推荐接口允许继续存在。
- 若需要平滑迁移，可将 V1 的选品、库存、文案能力逐步下沉为 worker 内部实现或工具实现。

### 10.3 不再采用的旧口径

以下表述在本修订后不再作为当前有效设计：

- “V1 Supervisor 是系统中心”
- “B 端与 C 端天然冲突”
- “首版要启动真实 MCP server”
- “首版要实现正式 Scheduler 子系统”

## 11. 降级与错误恢复

### 11.1 分层恢复策略

#### Tool 级

- 参数校验失败：不重试，直接返回结构化错误
- 瞬时错误：最多 1 次重试
- 仍失败：返回 tool fallback 或失败结果

#### Worker 级

- 可恢复错误时最多 1 次重试
- 仍失败：返回默认结构化结果
- 记录到 TaskBoard

#### Manager 级

- 信息不足：`needs_clarification`
- 非关键 worker 失败：尽量组织部分结果回复
- 关键路径失败：`fallback_used`

#### Background 级

- 推荐刷新任务最多 3 次重试
- 仍失败：事件进入 `failed`
- 保留错误信息，避免 silent failure

### 11.2 常见失败场景

| 场景 | 处理 |
|------|------|
| scene 缺少必需上下文 | `clarify` 或回退 `default` |
| 非法 tool / 非法 worker | `reject` |
| 偏好抽取失败 | 继续保留会话，但不更新长期画像 |
| 库存查询失败 | 返回无库存补充信息的候选结果，或直接降级 |
| 文案生成失败 | 返回商品结构化结果，无文案或默认文案 |
| 推荐刷新失败 | 事件转 `failed`，旧快照可继续服务 |

### 11.3 故障隔离

```text
对话失败     → 推荐快照读取仍可正常工作
后台刷新失败 → 旧快照仍可对外服务
画像更新失败 → session memory 仍保留，不直接丢失用户偏好
单个 worker 失败 → 不得导致整个进程崩溃
```

## 12. 测试验收

### 12.1 功能测试

1. 多轮对话能抽取预算、品类、品牌、用途、排除项。
2. 默认 advisory 消息返回文本回答与偏好抽取结果，而不是直接给低质量推荐。
3. worker 之间没有直接通信，所有任务都经过主 manager。
4. feedback 事件会入库，并在 V2.2 中以“homepage refresh 请求”的方式驱动最小闭环；但它不会直接在线改写 `UserProfile` 主字段。

### 12.2 平台测试

5. `Policy Gate` 能正确区分 `allow / reject / clarify`。
6. Hook Bus 能在核心事件点触发。
7. PromptRegistry 能按 `name + version` 正确加载模板。
8. TaskBoard 能同时追踪会话任务和后台任务。

### 12.3 闭环测试

9. 偏好稳定后会先入队 `profile_projection`，而不是同步更新 `user profile`。
10. `profile_projection` 消费后会生成 `user profile`，并继续触发 `homepage recommendation_refresh`。
11. 后台事件处理后能生成新的 `homepage` 推荐快照；失败时旧快照继续可读。
12. `homepage`、`default`、`product_page`、`cart` 的纯读 fallback 规则被正确执行。

### 12.4 持久化测试

13. 服务重启后，会话、画像、快照、事件、任务记录仍可恢复。
14. 未达到稳定偏好的对话不会污染长期画像。
15. 失败事件不会 silently disappear。

### 12.5 文档一致性检查

文档读完后应能直接回答：

- 系统有哪些 agent 层次
- 主 manager 和 worker 的通信边界是什么
- Runtime 包含哪些内核子系统
- 推荐闭环怎么打通
- 哪些是首版实现，哪些只是预留

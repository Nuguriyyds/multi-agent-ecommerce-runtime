# 智能电商推荐系统

## 产品目标

**一句话解释：**让电商平台的**推荐、文案、库存**三个能力协同工作，为每位用户生成个性化的、可履约的推荐结果。

**解决了什么问题：**传统电商存在三大痛点

| 痛点 | 传统做法 | 本项目做法 |
|------|---------|---------|
| 推荐结果和库存脱节 | 推荐了缺货商品 | 库存实时校验，缺货自动剔除 |
| 营销文案千篇一律 | 所有人看同一段广告语 | 根据用户画像生成个性化文案 |
| 各系统各自为战 | 推荐、文案、库存三套系统互不感知 | 统一编排，结果实时互相影响 |

## 用户角色

| 角色 | 怎么用 | 关心什么 |
|------|--------|---------|
| **终端消费者** | 浏览商品时，系统自动返回个性化推荐+文案 | 推荐准不准、文案有没有吸引力 |
| **运营人员** | 通过 API 或日志查看推荐效果、配置 A/B 实验参数 | 点击率、转化率、能否按人群调策略 |
| **开发者** | 通过 API 接入推荐能力，集成到现有电商系统 | 接口简单、响应快、结果稳定 |

## 功能范围

**核心能力：**
- 用户画像：实时提取用户行为特征（浏览/点击/购买），进行用户分群，输出结构化画像标签
- 商品推荐：基于用户画像从商品库中召回候选商品，排序后输出 Top-N 推荐列表
- 营销文案：根据用户画像 + 推荐商品，用 LLM 生成个性化营销文案
- LLM 重排（可选增强）：默认走传统排序，可通过 A/B 实验开关启用 LLM 对召回结果重排序，验证其对推荐效果的收益
- 库存校验：实时查询推荐商品的库存状态，剔除缺货/低库存商品，避免推荐无法履约的商品
- 统一编排：上述四个能力支持并行调度、聚合结果后统一返回

**冷启动策略：**
- 新用户/无画像用户：退回热门商品 + 当前活动池，不依赖个性化画像
- 低活跃用户：基于有限行为做粗粒度分群（如仅靠注册时选择的品类偏好）

**关键链路 vs 可降级链路：**

| 能力 | 失败处理 | 原因 |
|------|---------|------|
| 库存校验 | **强校验**：失败时走缓存兜底，缓存也不可用则该商品不出推荐列表 | 推荐不可售商品是业务事故 |
| 用户画像 | 可降级：失败时走冷启动策略 | 没画像也能推荐，只是不够精准 |
| 商品推荐 | 不可降级：失败则整个请求返回错误 | 这是核心功能 |
| 营销文案 | 可降级：失败时返回商品默认描述 | 没有个性化文案不影响推荐本身 |

**实验与观测：**
- A/B 测试：支持按用户分桶，对不同策略进行对比实验，演示环境使用模拟点击数据验证
- 效果日志：记录每次推荐的响应时间、各能力模块耗时、推荐结果命中情况，通过 API 可查询

**不做的事（明确边界）：**
- 不做用户注册/登录/订单/支付等电商基础功能
- 不做前端页面，只提供 API（运营通过日志/报表接口查看效果）
- 不做模型训练，使用现成 LLM 服务

## 预期结果

**功能维度：**
- 输入一个用户 ID，返回个性化推荐列表 + 每个商品的定制文案，且所有推荐商品均有库存
- 输入一个无历史行为的用户 ID，返回热门/活动商品 + 通用文案（冷启动）
- 支持通过 API 调用，响应格式为结构化 JSON

**性能维度：**
- 端到端响应时间 P99 < 3s（各能力模块并行执行）
- 关键链路（库存）有缓存兜底，非关键链路（文案）失败不阻塞主流程

**可验证的演示场景：**
- 同一商品，不同用户画像 → 生成不同风格的营销文案
- 某商品库存为 0 → 自动从推荐列表中剔除
- 新用户无画像 → 返回热门推荐而非报错
- A/B 测试：同一批用户，策略 A vs 策略 B → 可通过日志对比模拟点击率差异

## 关键决策记录
- LLM 只用于文案生成 + 可选重排（通过 A/B 开关），不参与主推荐决策
- 库存校验为关键链路（强校验 + 缓存兜底），文案为可降级链路
- 首期外部服务（Redis/Milvus）用 mock，后续按需接入真实服务
- 演示环境使用模拟点击数据验证 A/B 效果
- Python 环境注意：系统有 Python 3.11 和 3.13 两个版本，pip 默认装到 3.13，需用 python -m pip 确保装到 3.11；启动用 python -m uvicorn
- BaseAgent 设计：超时直接降级不重试（超时通常是下游服务整体不可用，重试无意义）；普通异常走指数退避重试
- AgentResult 用 Pydantic 模型统一包装（success/data/degraded/attempts/error/latency_ms），方便下游 Supervisor 编排判断
- F03 以 设计.md 为技术边界实现，不额外扩展未列出的能力；LLM API key 为空时走确定性 mock 分析，真实 key 存在时走 OpenAI 兼容接口
- F03 冷启动策略不调用 LLM，直接返回 `cold_start=true`、热门品类标签和 `new_user` segment
- F03 的 LLM 非法 JSON 视为普通异常，沿用 BaseAgent 重试语义；重试耗尽后降级为 cold start
- F04 当前只实现规则排序链路：无 profile 或 `cold_start=true` 时走粗召回，有 profile 时基于品类偏好 / 价格区间 / 标签 / segment 做重排；LLM 重排保留到 F10
- Product 领域模型内部字段使用 `product_id`，对外序列化为 `id`，这样既满足当前 API/schema 验收，也给后续库存校验保留稳定主键
- F05 将库存状态和过滤后商品一起返回；缓存兜底只返回“已有库存校验结果”的商品，避免未验证商品继续进入下游链路
- F06 文案模板优先级为 `new_user > high_value > churn_risk > price_sensitive > active`，同一用户命中多个 segment 时按该顺序选择单一模板，避免风格互相污染
- F06 的 LLM mock 扩展为同时支持画像分析和文案生成；文案链路统一使用 `marketing_copy_input` JSON 载荷，保证无外部依赖时也能稳定测试不同风格输出
- F06 合规过滤放在 Agent 内部收口，先接收 LLM 原始文案，再对“最好”“100%”等敏感词做替换；如果 LLM 超时或返回不可用 JSON，则整批降级为商品默认描述
- F07 的 Supervisor 固定为 3 阶段 DAG：`画像 + 粗召回` 并行、`精排 + 库存` 并行、`文案` 串行；粗召回候选数固定为 `num_items * 3`
- F07 最终推荐结果始终以库存校验结果为准：输出商品 = `精排顺序 ∩ 已校验且可售商品`，不允许未校验商品穿透到最终响应
- F07 优先复用画像 Agent 的降级 cold start profile；如果画像 payload 本身不可解析，再回退到无画像推荐路径
- F07 文案结果按最终商品 ID 对齐；如文案缺失则补商品默认描述，确保 `copies` 与 `recommendations` 一一对应
- F08 对“同一用户固定分桶”和“动态流量调配”采用分离策略：用户首次命中实验时基于 `MD5(experiment:user_id)` 取稳定 bucket，再按当下 Thompson Sampling 估算流量区间分配 variant；一旦分配后写入内存 assignment，后续调用不漂移
- F08 默认实验 `rec_strategy` 直接产出后续 F10 所需策略参数：`control` 返回 `rerank_enabled=false`，`treatment` 返回 `rerank_enabled=true`
- F08 Thompson Sampling 使用标准库 `random.betavariate` 和 Beta 后验（`alpha=prior+successes`, `beta=prior+failures`），避免引入 numpy 等额外依赖
- F10 的 LLM 重排只改变候选商品顺序，不改变 `Product` 输出结构；重排后沿用规则排序阶段生成的递减 score 序列，保证 treatment 路径和 control 路径的 schema 以及排序字段语义保持一致
- F10 若 LLM 返回无效结果，则在 `ProductRecAgent` 内部直接回退到规则排序结果而不是整条推荐链路降级；这样 treatment 组仍然稳定返回可用推荐，同时将 LLM 重排视为可选增强而非核心依赖
